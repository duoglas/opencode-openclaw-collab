## 2. 关键E2E用例

### E2E-01：标准正向流程
- **目的**：验证主路径可用性与结果正确性。
- **前置条件**：上下游服务健康，配置使用生产等价参数。
- **步骤**：
  1. 发送合法请求（覆盖主要业务参数组合）。
  2. 观察处理链路：入口→业务处理→下游调用→落库/回执。
  3. 校验响应码、响应体、数据持久化一致性。
- **断言**：全链路成功，业务状态一致，关键字段不缺失。

### E2E-02：成功率基线（30分钟稳态）
- **目的**：验证端到端成功率≥99%。
- **步骤**：
  1. 以稳定负载运行30分钟。
  2. 记录总请求数、成功数、失败数、超时数。
  3. 统计成功率并按失败类型分类。
- **断言**：成功率≥99%，且失败样本有可追踪日志与错误码。

### E2E-03：P95时延达标
- **目的**：验证P95<200ms。
- **步骤**：
  1. 按阶梯升压执行5-10分钟/档。
  2. 采集端到端时延分位（P50/P90/P95/P99）。
  3. 对比不同RPS档位。
- **断言**：目标业务档位下P95<200ms，且无异常抖动尖峰持续化。

### E2E-04：重试策略生效
- **目的**：验证失败自动重试（含退避）正确执行。
- **步骤**：
  1. 下游注入瞬时失败（5xx/超时）。
  2. 观察重试次数、间隔、退避曲线。
  3. 统计重试后成功率。
- **断言**：重试参数与设计一致；可恢复失败在重试后成功。

### E2E-05：DLQ转移与可观测
- **目的**：验证重试耗尽后消息进入DLQ。
- **步骤**：
  1. 注入持续失败场景（不可恢复）。
  2. 触发重试耗尽。
  3. 校验DLQ消息内容、来源、错误上下文。
- **断言**：DLQ入队准确，无漏转、错转；支持后续人工/自动补偿。

### E2E-06：幂等一致性
- **目的**：验证重复请求不会重复生效。
- **步骤**：
  1. 使用同一幂等键重复提交（串行10次、并发50次）。
  2. 检查最终业务状态与副作用（写库、事件投递）。
- **断言**：仅一次生效，重复请求返回一致结果或幂等命中标识。

### E2E-07：断链恢复
- **目的**：验证链路中断后的恢复能力。
- **步骤**：
  1. 人工断开关键依赖（MQ/DB/下游网络）5分钟。
  2. 恢复连接并持续观测30分钟。
  3. 校验积压处理、重试回收、数据完整性。
- **断言**：系统恢复后可持续稳定运行，成功率回归≥99%，无不可恢复数据丢失。

## 3. 压测方法与指标采集

### 3.1 压测方法
- **模型**：
  - 基线稳态：固定RPS（如100 RPS）30分钟。
  - 阶梯升压：每10分钟提升一档（50→100→200→400 RPS）。
  - 峰值冲击：短时突发（2~3倍目标流量，持续3分钟）。
- **流量构成**：
  - 80%常规请求，15%边界参数请求，5%异常请求。
  - 幂等键重复流量单独注入（并发冲突场景）。
- **故障注入**：
  - 下游超时、5xx、连接拒绝、网络抖动。
  - 中断与恢复组合演练（用于断链恢复验证）。

### 3.2 指标采集
- **请求级指标**：QPS/RPS、成功率、错误率、超时率、重试触发率、DLQ入队率。
- **时延指标**：P50/P90/P95/P99、最大时延、时延抖动。
- **系统资源**：CPU、内存、GC、线程/连接池、水位、磁盘I/O、网络吞吐。
- **队列与恢复**：队列积压深度、消费延迟、恢复后清 backlog 时间。
- **幂等与数据质量**：重复写入率、唯一约束冲突数、状态不一致数。

### 3.3 采集与判定口径
- 采样粒度：1s（实时），1m（聚合看板）。
- 统计窗口：稳态窗口不少于30分钟。
- 判定口径：
  - 成功率 = 成功请求数 / 总请求数。
  - P95以端到端总时延计算（入口到最终可见结果）。
  - DLQ准确率 = 实际DLQ条数 / 应入DLQ条数。

## 4. 验收清单与签核条件

### 4.1 验收清单
- [ ] E2E主流程全部通过，无阻断级缺陷。
- [ ] 端到端成功率 **≥99%**（稳定窗口≥30分钟）。
- [ ] 端到端时延 **P95<200ms**（目标业务负载下）。
- [ ] 重试策略生效且参数符合设计（次数/间隔/退避）。
- [ ] 重试耗尽后进入DLQ，消息可追踪、可补偿。
- [ ] 幂等场景下无重复生效、无重复写入。
- [ ] 断链恢复演练通过，恢复后成功率回归并清空积压。
- [ ] 监控看板、日志与告警规则可用，留存完整测试证据。

### 4.2 签核条件（Go/No-Go）
- **Go（准入）**：
  1. 上述清单全部勾选通过；
  2. 无P0/P1未关闭缺陷；
  3. P2缺陷有明确缓解与上线后追踪方案；
  4. 压测报告、E2E报告、故障演练记录齐全可追溯。
- **No-Go（不准入）**：
  - 任一硬性指标不达标（成功率/P95/幂等/DLQ/断链恢复）；
  - 监控告警不可用或证据链不完整。

### 4.3 签核角色建议
- 刑部（测试负责人）：测试执行结果确认。
- 工部（研发负责人）：缺陷关闭与修复有效性确认。
- 运维/平台：压测环境与监控数据真实性确认。
- 业务代表：关键业务路径可用性确认。
