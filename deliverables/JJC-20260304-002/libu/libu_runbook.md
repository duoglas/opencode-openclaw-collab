# 六部联动发布运行手册（礼部版）

> 适用范围：本手册用于支撑“六部实施推进”相关项目的**发布执行、快速故障排查、回滚决策与执行**以及**演练复盘**。
> 角色假定：技术（工部）、运维（兵部）、产品/业务（户部）、风控/质检（刑部）、对外沟通（礼部）、统筹/稽核（吏部）。

---

## 一、发布前准备清单（T-1 天及之前）

### 1.1 发布决策与范围确认

- [ ] 发布目标与收益已在方案中明确，且经六部评审通过
- [ ] 发布内容已冻结（功能列表、配置项、数据变更、脚本）
- [ ] 变更影响面评估完成：
  - [ ] 系统/服务影响清单
  - [ ] 依赖方与被依赖方清单
  - [ ] 性能与容量评估
  - [ ] 安全与合规评估
- [ ] 是否允许当日回滚策略已达成一致（回滚窗口、回滚代价）

### 1.2 技术与环境准备

- [ ] 代码已合并至发布分支，CI 通过，无红灯
- [ ] 关键单测、集成测试通过率满足门槛
- [ ] 灰度/预发环境验证完成，结果记录在案
- [ ] 数据变更脚本已在影子/测试环境演练
- [ ] 监控、日志、告警规则已覆盖新增/变更模块
- [ ] 容量预估与压测结果满足上线要求

### 1.3 人员与沟通准备

- [ ] 发布负责人（总指挥）已指定
- [ ] 各部关键联系人、电话/IM 群确认无误
- [ ] 发布时间窗口已与业务确认并公告
- [ ] 游客/客户侧公告口径草案准备就绪（含失败/回滚口径）

### 1.4 回滚与应急预案准备

- [ ] 回滚策略已书面化（见下文第 3 章）并经评审
- [ ] 关键数据备份方案已验证可用
- [ ] 回滚所需脚本、配置、版本包已准备并通过演练
- [ ] 明确“不可回滚项”（如：不可逆数据脚本）及替代处置方案

## 二、发布执行手册

### 2.1 发布前 30 分钟

- [ ] 再次确认参与人在线与沟通通道畅通（专用群 / 电话桥）
- [ ] 核查监控面板、告警系统处于正常状态
- [ ] 确认当日其他变更窗口冲突情况（避免叠加风险）
- [ ] 宣读本次发布范围、目标、回滚边界

### 2.2 标准发布步骤（示例，可按实际系统调整）

1. **冻结变更**：停止非本次发布相关变更的合入与上线
2. **只读/限流**（如需要）：
   - 对关键写入路径实施只读或限流
   - 对长事务任务进行收敛
3. **备份关键数据/配置**：
   - 配置中心、数据库关键表、KV 缓存关键前缀
4. **部署发布包**：
   - 按服务/模块分批滚动发布
   - 每批发布后观察监控与日志不少于 5 分钟
5. **执行数据/配置变更**：
   - 采用“先暗后明”原则：先加配置开关，再逐步放量
6. **健康检查与业务验证**：
   - 核心接口探活
   - 核心业务路径拨测
   - 重要报表/后台页面抽查
7. **解除只读/限流**（如有）
8. **更新变更记录**：
   - 记录实际执行时间、操作者、版本号、关键操作 ID

### 2.3 发布完成后的观察期

- [ ] 重点监控指标在观察期（建议 30–60 分钟）内无异常
- [ ] 客户投诉、客服工单无异常聚集
- [ ] 日志中错误率、超时率符合预期
- [ ] 若存在灰度放量，放量节奏、阈值提前约定并执行记录

## 三、回滚预案（含 5 分钟内动作）

### 3.1 回滚触发标准

满足以下任一条件，应立即进入应急回滚评估（不争辩、先止血）：

- 关键交易指标下降 ≥ X%（由业务预设）且持续 Y 分钟
- 错误率、超时率明显高于基线并影响主路径
- 安全、合规、资损风险事件已发生或高度疑似
- 客户/合作伙伴侧重大体验问题，经业务确认无法短时规避

### 3.2 回滚决策角色

- 发布总指挥：有权发布“立即回滚”指令
- 技术负责人：评估技术可行性、操作路径
- 业务负责人：评估业务影响与窗口
- 风控/法务（如适用）：评估合规与资损

### 3.3 **触发后的黄金 5 分钟动作清单**

> 目标：在 5 分钟内完成“止血+决策+启动执行”。

**第 0–1 分钟：确认与止血**

- [ ] 立即冻结当前发布操作（停止继续放量/发布）
- [ ] 切换为最保守流控策略（必要时降级非核心功能）
- [ ] 技术负责人在监控/日志中快速定位“是否与本次变更强相关”

**第 1–3 分钟：快速决策**

- [ ] 总指挥召集关键人（技术、业务、运维）进行“1 分钟战情汇报”：
  - 当前现象与范围
  - 是否只影响增量用户/灰度批次
  - 临时缓解措施是否有效
- [ ] 根据预设触发标准，做“回滚 / 继续 / 暂缓观察”三选一决策
- [ ] 如决策为“回滚”，立即口头确认：
  - 回滚到哪个版本/状态
  - 是否包含数据回滚

**第 3–5 分钟：启动回滚与公告**

- [ ] 运维/技术按预案启动回滚脚本或回滚流程
- [ ] 礼部/对外沟通在专用群内发出“应急播报”（模板见下文）
- [ ] 总指挥确认“预计恢复时间（ETA）”并同步

### 3.4 标准回滚步骤（技术向）

1. 停止当前版本新实例的流量接入
2. 按顺序将流量切回上一稳定版本
3. 如涉及数据库：
   - 优先关闭新增高风险写入
   - 执行可逆数据变更的回滚脚本
   - 对不可逆变更执行补偿/修复措施
4. 清理缓存、刷新配置，避免旧版本读取新结构导致错误
5. 回滚完成后进行健康检查与业务验证
6. 在变更记录中详细记录回滚原因、耗时、范围

### 3.5 回滚后的复盘要点

- 回滚触发是否及时？是否存在“犹豫时间”
- 5 分钟动作是否按预案执行，有无临时决策
- 监控与告警是否足够敏感与准确
- 回滚过程是否暴露新的风险（如脚本不可靠）

## 五、故障排查指引（发版相关）

> 目的：在发布过程中或发布后出现异常时，提供结构化排查路径，避免“盲目抓日志”。

### 5.1 快速判断：是否与本次发布相关

1. 时间维度：异常是否从发布窗口开始出现？
2. 范围维度：
   - 仅影响灰度/新版本节点？
   - 老版本是否同样异常？
3. 功能维度：是否集中在本次变更涉及的模块/接口？
4. 回滚验证：在小范围回滚/摘除某节点后，异常是否缓解？

如以上 2 项及以上指向本次变更，则视为“高度相关”，优先考虑回滚而非长时间排查。

### 5.2 排查步骤（建议顺序）

1. **监控总览**：
   - QPS、RT、错误率、资源使用率（CPU、内存、磁盘、带宽）
   - 业务关键指标（转化率、成功率、UV、GMV 等）
2. **降维定位**：
   - 是否集中在特定服务/实例
   - 是否特定地域/渠道/终端类型
3. **日志与链路追踪**：
   - 抽样失败请求的 TraceID 进行链路追踪
   - 对比发布前后日志差异（错误码、异常堆栈）
4. **配置与依赖检查**：
   - 配置中心是否发布了新配置
   - 外部依赖（支付、三方接口、消息队列、缓存）是否异常
5. **数据库与数据层**：
   - 慢查询、锁等待、死锁情况
   - 表结构或索引变更是否生效异常
6. **前端与客户端**（如适用）：
   - 版本是否强制更新导致兼容问题
   - 是否存在缓存/版本混用

### 5.3 常见问题分类与处置建议

- 性能问题：优先开启限流/降级，减轻系统压力，再寻找热点与慢点
- 功能缺陷：尽量通过配置开关关闭有问题的新功能，等待修复版本，而不是立即回滚全部
- 数据问题：优先保护数据安全，暂停写入，再评估回滚或补偿
- 外部依赖异常：先确认第三方状态，必要时切换备用通道或降级

## 六、演练计划与演练报告模板

### 6.1 演练类型

- **桌面演练**：围绕假设场景在会议室推演各角色决策与动作
- **沙箱演练**：在非生产环境完整执行发布+回滚流程
- **生产演练（谨慎）**：在低风险窗口、小流量灰度下验证预案

### 6.2 演练准备要点

- 明确演练目标（检验流程 / 检验脚本 / 检验监控与告警等）

- 预设演练中各关键时间点（发现、决策、回滚启动、恢复）
