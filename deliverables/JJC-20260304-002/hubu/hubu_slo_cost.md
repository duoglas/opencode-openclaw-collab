## 一、SLO 口径建议

六部整体可采用三层结构：**用户感知 SLO → 业务域 SLO → 基础设施 SLO**。

### 1. 用户感知 SLO（自上而下的“体验口径”）

- **成功率 SLO**：
  - 口径：在统计周期内，用户发起的有效请求中，最终获得正确业务结果的比例。
  - 公式：`成功率 = 成功完成请求数 / 有效请求总数`
  - 有效请求：排除明显参数错误、越权访问等“用户侧问题”，仅统计服务侧应当能正常处理的请求。
  - 建议目标：关键路径 ≥ 99.9%，一般路径 ≥ 99.5%。

- **P95 响应时间 SLO**：
  - 口径：在统计周期内，用户请求从**进入网关**到**收到完整业务响应**的 95 分位耗时。
  - 公式：对单请求耗时分布取 P95；可按接口/业务类型分别统计。
  - 建议目标：
    - 交互敏感型（控制台、管理后台）：P95 ≤ 800ms
    - API 调用型（对外接口）：P95 ≤ 500ms（或按现状 + 20% 收敛空间逐步优化）

- **可用性 SLO（按时间维度）**：
  - 口径：在统计周期内，系统处于“可服务”状态的时间占比。
  - 可服务定义：核心接口成功率 ≥ 99% 且 P95 ≤ 2s。
  - 计算粒度：可按分钟/5 分钟粒度打点聚合。

### 2. 业务域 SLO（部门/微服务维度）

- 按照六部职能，将服务划分为若干业务域（例：吏部账号与权限、户部计费与结算、兵部调度、刑部风控、工部基础设施、礼部通知等）。
- 每个业务域需要承诺：
  - 其对外暴露的**接口成功率 SLO**
  - 核心接口的 **P95/P99 延迟目标**
  - 自身**依赖下游失败时的降级策略 SLO**（例如：下游异常时 95% 请求仍能返回有价值的降级结果）。

### 3. 基础设施 SLO（底层资源与通道）

- 包括网关、消息队列、数据库、缓存、对象存储等。
- SLO 更偏向于：
  - **服务可达性**（连接成功率、握手失败率）
  - **资源使用率**（CPU、内存、连接数、磁盘 IO）
  - **内部延迟**（队列排队时间、连接建立时间等）

## 二、指标体系设计

围绕问题提出的 5 个关键指标：**成功率 / P95 / 重试命中率 / DLQ / 越权拒绝率**，建议统一定义如下。

### 1. 成功率

- **指标名**：`request_success_rate`
- **定义**：在统计周期内，所有有效请求中，返回“业务成功”结果的比例。
- **分母口径**：
  - 通过认证与鉴权的请求。
  - 不含明显参数错误（4xx 中的 400/422 等可单独记为 `client_error_rate`）。
- **分子口径**：
  - 返回业务成功码（如 `code=0` 或 HTTP 2xx 且无业务错误）的请求。
- **建议维度**：
  - 按业务域/服务名
  - 按接口/路由
  - 按调用方（client_id / app_id）
  - 按集群/机房

### 2. P95 响应时间

- **指标名**：`request_latency_p95`
- **定义**：统计周期内单请求耗时的 95 分位数。
- **分布来源**：
  - 建议使用直方图型指标（如 Prometheus Histogram），按接口或路由采集。
- **口径**：
  - 推荐采用“端到端”用户视角：入口网关接收时间 → 最终响应写出时间。
  - 对内部链路还可以拆分：网关耗时 / 应用逻辑耗时 / 下游依赖耗时。

### 3. 重试命中率

- **指标名**：`retry_success_rate`
- **定义**：在所有重试请求中，通过重试最终成功的比例。
- **口径拆解**：
  - `retry_total_count`：触发重试机制的总次数。
  - `retry_success_count`：重试后成功的次数。
  - 指标公式：`retry_success_rate = retry_success_count / retry_total_count`。
- **价值点**：
  - 反映系统对瞬时性故障（transient error）的自愈能力。
  - 过高可能意味着底层质量差（依赖不稳），过低则可能说明重试策略设置不合理。

### 4. DLQ 指标

- **指标名**：`dlq_message_rate`, `dlq_backlog_size`
- **定义**：
  - `dlq_message_rate`：单位时间内进入死信队列（DLQ）的消息数量或比例。
  - `dlq_backlog_size`：当前积压在 DLQ 中的消息总数。
- **口径要点**：
  - 统计来源：消息中间件（如 Kafka/RabbitMQ/云 MQ）的 DLQ 主题或队列。
  - 可按业务类型或 topic 分维度：订单类消息、计费类消息、通知类消息等。
- **阈值建议**：
  - 针对关键业务设置告警：
    - `dlq_message_rate` 高于基线 X 倍
    - 或 `dlq_backlog_size` 连续 N 个周期不下降

### 5. 越权拒绝率

- **指标名**：`unauthorized_reject_rate`
- **定义**：在所有到达系统的请求中，由于鉴权/鉴权失败（无权限、越权访问）而被拒绝的比例。
- **分母口径**：
  - 所有进入网关的请求（可含明知会失败的探测流量）。
- **分子口径**：
  - 返回 401/403，或业务侧判断为越权访问的请求数。
- **注意区分**：
  - 与“系统异常导致失败”分开，避免混淆：
    - `unauthorized_reject_rate`：安全策略正常工作的一部分，可维持在一定水平。
    - `security_error_rate`：真正的安全系统异常（例如鉴权服务不可用导致误拒绝合法请求）。

### 6. 指标间的关系与仪表盘布局建议

- **核心服务总览**：
  - 时间维度：最近 1h / 6h / 24h。
  - 卡片：成功率（SLO 对比线）、P95/P99 延迟、当前错误率（按类型拆分）。
- **重试与自愈视图**：
  - 重试总次数、重试成功率、重试带来的额外 QPS 比例。
- **队列与 DLQ 视图**：
  - 各 topic 的消费延迟、积压量、DLQ rate/backlog。
- **安全视图**：
  - 越权拒绝率趋势，按调用方/接口分布。

## 三、容量成本模型

目标：支持六部在“扩容/降配/限流/多活”决策时，有**统一的成本视角**。

### 1. 基本变量

设：

- `Q`：平均请求量（QPS 或日调用数）。
- `L_p95`：P95 延迟目标。
- `S`：成功率目标（如 99.9%）。
- `C_unit`：单位资源成本（例如单节点单小时成本，或单 Pod/单容器成本）。
- `N`：实例数量（或资源配额折算的“虚拟实例数”）。
- `U_cpu` / `U_mem`：CPU/内存使用率。
- `C_fixed`：固定成本（基础设施、带宽保底等）。
- `C_var`：随流量变化的可变成本（节点、存储、按量计费等）。

### 2. 容量与性能关系（简化模型）

对于单一服务，可以采用排队模型的简化近似：

- 当 CPU 使用率 `U_cpu` 接近 70%–80% 时，P95 延迟会急剧恶化。
- 可定义：
  - **安全负载区**：`U_cpu ≤ 60%`，P95 延迟接近最佳。
  - **告警负载区**：`60% < U_cpu ≤ 80%`，P95 逼近 SLO 上界。
  - **危险区**：`U_cpu > 80%`，P95 及错误率急剧恶化。

在给定 QPS 和复杂度的情况下：

- `N_min`：满足当前 SLO 的最小实例数。
- `N_safe`：在 N_min 基础上增加冗余（例如 +30% 容量）以应对波动。

### 3. 成本计算

在任意时刻：

- `C_var = N * C_unit`
- `C_total = C_fixed + C_var`

可进一步按业务域拆分：

- 为每个业务域建立“预算标签”，统计其消耗的资源占总量比例。
- 支持：
  - 按服务/业务线查看成本占比
  - 对照其带来的请求量/收入/GMV，评估性价比

### 4. 容量规划建议

- **按 SLO 反推容量**：
  - 基于负载测试或历史数据，绘制 QPS → P95 曲线。
  - 找到满足 P95 目标的最大 QPS；据此估算单位实例承载能力。
  - 按预测峰值流量 + 冗余系数，得出需要的实例数 N_safe。

- **按成本约束优化**：
  - 在满足最低 SLO（例如成功率 99.5%，P95 1s）的前提下，尝试缩容。
  - 监控 P95 与错误率，一旦接近 SLO 下限，自动回滚扩容。

- **重试与容量耦合**：
  - 重试策略会放大流量：
    - 有效请求量 `Q_effective = Q * (1 + R_retry)`，其中 `R_retry` 是平均重试次数。
  - 在容量模型中需要计入重试流量对 N 的放大效应。

## 四、看板与治理建议

### 1. 看板结构

建议六部统一建设三层看板：

1. **集团总览看板**（面向决策层）：
   - 指标：各业务域核心 SLO 达标率（成功率、P95、可用性）。
   - 展示：
     - 红黄绿灯 + 趋势线
     - 当日重大事件/告警摘要

2. **业务域 SLO 看板**（面向各部堂官）：
   - 按服务/接口维度展示：
     - 成功率、P95/P99
     - 错误率拆解（5xx、业务错误、重试后仍失败）
     - 重试命中率
     - DLQ rate/backlog
     - 越权拒绝率（可按调用方排序，识别异常应用）

3. **运维与容量看板**（面向技术团队）：
   - CPU/内存利用率、连接数、队列积压、实例数
   - 单位 QPS 成本（例如：`cost_per_1k_requests`）
   - 资源预算执行情况（当月累计成本 vs 预算）

### 2. 治理节奏

- **日常**：
  - 每日例会前自动生成 SLO 达标日报（可用率、P95、主要告警）。
- **每周**：
  - 周会回顾：
    - SLO 违约次数和总违约时间
    - TOP N 故障根因
    - 重试与 DLQ 异常情况
- **每月/每季度**：
  - 成本回顾 + 资源优化：
    - 按业务线拆分资源占用与 SLO 目标达成情况
    - 找出“高成本/低达成”或“低成本/高达成”的业务进行结构优化

## 五、后续可落地动作建议

1. 统一埋点规范，固化上述指标名与标签（service、endpoint、caller、cluster 等）。  
2. 建立“基础指标 → SLO → 告警 → 看板”闭环（指标采集 → SLO 计算 → 告警规则 → 仪表盘）。  
3. 将容量成本模型接入预算体系，按月生成《六部资源消耗与 SLO 达标报告》。  
4. 逐步推进关键链路（支付、结算、风控、权限等）达到 99.9%+ 的用户感知 SLO，非关键链路按权衡设定 SLO 级别。
