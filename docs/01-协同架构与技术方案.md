# OpenCode × OpenClaw 多实例协同架构与技术方案（Tailscale + Telegram）

## 1. 背景与目标

### 已知约束
- 并发任务低（主要由单一人类发起）
- 不使用外网 Webhook
- 网络通信基于 Tailscale 内网
- 人机入口与通知使用 Telegram

### 核心目标
1. 通过 Telegram 统一下达任务与接收回报
2. 由 OpenClaw 统一编排多个 OpenCode 实例协同执行
3. 保证任务可追踪、可审计、可重试、可人工中断
4. 在低并发前提下保持实现简单、稳定、可扩展

---

## 2. 总体架构

采用：**单中枢（OpenClaw Router）+ 多执行节点（OpenCode Workers）**

```text
[Human @ Telegram]
        |
        v
[OpenClaw Router / Orchestrator]
   |        |         |
   v        v         v
[Worker-A] [Worker-B] [Worker-C]   (OpenCode instances over Tailscale)
   \        |        /
    \-------状态与结果回报------/

(通知统一回到 Telegram)
```

### 角色定义
- **OpenClaw Router（控制平面）**
  - 接收任务
  - 任务拆分与路由
  - 状态机管理
  - 失败重试
  - 最终汇总回报
- **OpenCode Worker（数据平面）**
  - 执行具体任务
  - 定期心跳
  - 返回结构化执行结果
- **Telegram（人机接口）**
  - 下达任务
  - 查看进展
  - 人工审批
  - 异常告警

---

## 3. 任务模型与消息协议

### 3.1 任务实体（Task Envelope）
每个任务使用统一结构：

```json
{
  "task_id": "TASK-20260304-001",
  "parent_id": null,
  "source": "telegram:7895363094",
  "priority": "P1",
  "type": "content|code|ops|analysis",
  "assignee": "worker-b",
  "dedupe_key": "sha1(normalized_request)",
  "timeout_sec": 900,
  "retry": {"max": 2, "backoff_sec": [30, 120]},
  "requires_approval": false,
  "payload": {
    "goal": "...",
    "constraints": ["no_webhook", "tailscale_only"],
    "deliverable": "markdown"
  }
}
```

### 3.2 状态机
`queued -> dispatched -> running -> done|failed|blocked|cancelled`

### 3.3 幂等规则
- `dedupe_key` 在 TTL（建议 24h）内只允许成功执行一次
- 重复提交返回“已受理/已完成”引用，不重复派发

---

## 4. 路由与分工策略（低并发优化）

### 4.1 固定路由优先（简单稳定）
- 文档/内容：Worker-LIBU
- 代码实现：Worker-GONGBU
- 审核测试：Worker-XINGBU
- 运维/健康：Worker-BINGBU

### 4.2 回退路由
若目标 Worker 无心跳 > 120s：
1) 转派同类备份 Worker
2) 无备份则回退到 OpenClaw 默认 agent
3) 同时发 Telegram 告警

---

## 5. 通信机制

### 5.1 控制信道
- Telegram -> OpenClaw（指令）
- OpenClaw -> Telegram（摘要、告警、审批请求）

### 5.2 机器协作信道
- OpenClaw <-> OpenCode via Tailscale 内网
- 禁止 Worker 之间绕过中枢直接互发任务（避免失控）

### 5.3 心跳与健康
- Worker 每 60s 心跳一次（简短）
- 失联阈值：120s
- 失联任务自动标记 `blocked` 并触发重派

---

## 6. 安全设计

1. **零公网暴露**：无外网 webhook，仅 Tailscale 网络访问
2. **最小权限**：Worker 按职责限制工具与目录
3. **高风险动作闸门**：删除/外发/生产变更必须审批
4. **凭证隔离**：每 Worker 使用独立凭证作用域
5. **审计留痕**：记录 task_id、执行者、关键操作、产出路径

---

## 7. 失败处理与恢复

### 7.1 重试策略
- 可重试错误：网络抖动、临时超时
- 不可重试错误：参数非法、权限拒绝、逻辑校验失败
- 重试节奏：30s、120s（最多2次）

### 7.2 人工接管条件
- 连续失败 >= 2
- 关键任务超时 >= 1 次
- 审批超时 >= 30 分钟

触发后统一 Telegram 通知：
- 任务ID
- 当前状态
- 最近错误
- 建议处置（重试/改派/终止）

---

## 8. 可观测性与运维

### 指标（建议最小集）
- 任务吞吐（tasks/day）
- 成功率（done / total）
- 平均耗时（MTTR）
- 重试率
- 人工介入率

### 日志分层
- Router 日志：任务生命周期
- Worker 日志：执行细节
- 审计日志：审批与高风险动作

---

## 9. 部署方案（分阶段）

### Phase 1（本周）
- 上线单 Router + 2~3 Worker
- 开启统一 task_id 与状态机
- Telegram 只做入口/回报

### Phase 2（下周）
- 增加重试与失败告警自动化
- 增加 dedupe 与 TTL 缓存
- 形成周报（成功率/失败原因）

### Phase 3（按需）
- 增加 Router 备份
- 增加任务优先级队列与SLA
- 精细化权限策略

---

## 10. 实施清单（可直接开工）

1. 定义统一任务 JSON 协议（task_id/dedupe_key/timeout/retry）
2. 在 OpenClaw 中实现 Router 路由规则（按类型分派）
3. 为每个 OpenCode Worker 绑定职责与最小权限目录
4. 接入心跳检查与 120s 失联重派逻辑
5. 建立 Telegram 模板消息（接单/进度/完成/告警）
6. 启用审计日志（任务、审批、高风险动作）
7. 先跑 1 周灰度，按失败类型调优

---

## 11. 结论

在“低并发 + 内网 + Telegram 控制”的前提下，
**最有效架构是：OpenClaw 统一编排，OpenCode 专职执行，Telegram 只做人机入口和告警。**

此方案复杂度低、可控性高、可审计，并可逐步演进到更高可用。
