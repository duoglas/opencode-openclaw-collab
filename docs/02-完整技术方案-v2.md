# OpenCode × OpenClaw 协同系统完整技术方案（v2）

> 场景约束：
> - 低并发（主要由单一用户发起）
> - 不走外网 Webhook
> - 基于 Tailscale 内网通信
> - Telegram 作为统一人机入口与通知通道
> - 存在较多代码开发/自动化执行任务

---

## 1. 建设目标与范围

## 1.1 目标
1. 统一任务入口：用户可通过 Telegram 或任一实例发起任务。
2. 统一编排：由 OpenClaw 作为中枢进行任务路由、拆分、执行协调。
3. 多实例协作：多个 OpenCode Worker 按职责执行（代码、文档、测试、运维）。
4. 稳定可靠：支持幂等、重试、超时接管、失败告警、人工审批。
5. 可审计：关键动作可追踪（谁在何时对什么资源执行了什么）。

## 1.2 非目标
- 不追求互联网级大规模并发。
- 不建设复杂分布式一致性系统（如 Raft/Kafka 大集群）。
- 不允许 Worker 间自由 P2P 指挥（避免失控）。

## 2. 架构拓扑设计

## 2.1 总体拓扑（逻辑）

```text
                        ┌───────────────────────────┐
                        │        Telegram           │
                        │  (命令入口/进度通知/审批) │
                        └─────────────┬─────────────┘
                                      │
                                      ▼
                    ┌──────────────────────────────────┐
                    │ OpenClaw Router / Orchestrator  │
                    │ - API Gateway                   │
                    │ - Task Scheduler                │
                    │ - Workflow Engine               │
                    │ - Approval & Policy Engine      │
                    │ - State Store                   │
                    └──────────┬───────────┬──────────┘
                               │           │
                 Tailscale Mesh│           │Tailscale Mesh
                               ▼           ▼
                  ┌────────────────┐   ┌────────────────┐
                  │ OpenCode-W1    │   │ OpenCode-W2    │
                  │ (coding主力)   │   │ (文档/内容)     │
                  └────────────────┘   └────────────────┘
                               │           │
                               ▼           ▼
                  ┌────────────────┐   ┌────────────────┐
                  │ OpenCode-W3    │   │ OpenCode-W4    │
                  │ (测试审计)      │   │ (运维/健康)     │
                  └────────────────┘   └────────────────┘
```

## 2.2 物理部署建议
- **控制节点（1台）**：OpenClaw Router + 状态存储（SQLite/Postgres）
- **执行节点（2~4台）**：各 OpenCode Worker（可同机多实例）
- **网络**：全部加入同一 Tailnet；只开放 Tailnet 内访问端口

## 3. 功能模块设计

## 3.1 控制平面模块（OpenClaw）

### A. Ingress 网关
- 接收 Telegram 指令/实例内部任务。
- 对输入做标准化（任务标题、目标、约束、优先级）。

### B. Task Manager（核心）
- 生成 `task_id`、`dedupe_key`。
- 维护任务状态机与重试计数。
- 处理父子任务（主任务拆分为子任务）。

### C. Router
- 按 `task.type` 与能力标签分发 Worker。
- 支持主备 Worker 回退。

### D. Workflow Engine
- 支持单任务、串行 DAG、并行子任务。
- 汇总子任务输出后生成统一回报。

### E. Approval Engine
- 高风险动作（删除/外发/生产写操作）进入审批。
- 审批通过后继续；超时自动取消或改派。

### F. Policy Engine
- 限制 Worker 工具白名单、目录边界、命令风险级别。

### G. Observability
- 任务事件日志、失败原因分类、执行时长统计。
- 告警推送至 Telegram。

## 3.2 数据平面模块（OpenCode Worker）

### A. Worker Agent Runtime
- 拉取/接收任务，执行后回传结构化结果。

### B. Capability Adapter
- 声明能力标签：`coding` / `doc` / `qa` / `ops`。

### C. Sandbox + FS Guard
- 限制可读写路径，避免越权访问。

### D. Heartbeat Reporter
- 每 60s 心跳；失联 >120s 触发接管。

### E. Result Reporter
- 上报 `output_artifacts`、`summary`、`metrics`、`next_actions`。

## 4. 任务与工作流模型

## 4.1 Task Envelope

```json
{
  "task_id": "TASK-20260304-001",
  "parent_id": null,
  "source": "telegram:7895363094",
  "type": "coding",
  "priority": "P1",
  "assignee": "worker-coding-1",
  "status": "queued",
  "dedupe_key": "sha256(...)" ,
  "timeout_sec": 1800,
  "retry": {"max": 2, "strategy": "exp", "backoff_sec": [30,120]},
  "approval": {"required": false, "level": "none"},
  "payload": {
    "goal": "实现功能X",
    "constraints": ["workspace_only", "no_external_webhook"],
    "deliverable": ["code", "tests", "report.md"]
  }
}
```

## 4.2 状态机
`queued -> dispatched -> running -> done|failed|blocked|cancelled`

### 状态切换关键规则
- `running -> blocked`：等待审批/外部依赖。
- `failed -> queued`：可重试错误且次数未超限。
- `failed -> cancelled`：不可重试或超过重试次数。

## 4.3 幂等与去重
- 同 `dedupe_key` 在 24h TTL 内只允许一个“成功执行实例”。
- 重复请求返回引用已有任务（避免重复执行开发任务）。

## 5. 接口设计

## 5.1 控制平面 API（建议内部 HTTP/JSON）

### 1) 创建任务
`POST /api/v1/tasks`

Request:
```json
{
  "source": "telegram:7895363094",
  "type": "coding",
  "priority": "P1",
  "payload": {"goal":"...","deliverable":"..."}
}
```

Response:
```json
{"task_id":"TASK-20260304-001","status":"queued"}
```

### 2) 查询任务
`GET /api/v1/tasks/{task_id}`

### 3) 领取任务（Worker Pull 模式）
`POST /api/v1/workers/{worker_id}/claim`

### 4) 心跳
`POST /api/v1/workers/{worker_id}/heartbeat`

### 5) 结果上报
`POST /api/v1/tasks/{task_id}/result`

### 6) 审批回调（Telegram 按钮触发）
`POST /api/v1/tasks/{task_id}/approval`

## 6. 关键技术细节设计

## 6.1 调度策略
- 默认：能力匹配 + 最小负载。
- 同时支持：固定路由（稳定优先，适合低并发）。

## 6.2 超时与接管
- 任务超时：`timeout_sec` 到期后发中断信号。
- Worker 失联：>120s 无心跳，任务回收重派。

## 6.3 错误分类
- `TRANSIENT`：网络抖动、短时超时（可重试）
- `PERMANENT`：参数错误、权限拒绝（不可重试）
- `HUMAN_REQUIRED`：审批缺失、业务歧义（人工介入）

## 6.4 审批模型
高风险动作需人工确认：
- 删除文件/目录
- 外发消息到第三方
- 生产环境部署或关键配置变更

审批超时策略：30分钟自动驳回并告警。

## 6.5 安全模型
- Tailscale ACL：仅允许控制节点访问 Worker 管控端口。
- Worker 最小权限运行（非 root，限制目录）。
- 凭证按 Worker 隔离，不共享高权限 Token。

## 6.6 可观测性
- 指标：成功率、平均时长、重试率、人工介入率。
- 日志：
  - 事件日志（状态切换）
  - 执行日志（命令/产物）
  - 审计日志（审批与高风险动作）

## 7. Telegram 交互设计

## 7.1 命令模板
- `/task <目标>`：创建任务
- `/status <task_id>`：查状态
- `/approve <task_id>`：批准待审任务
- `/cancel <task_id>`：终止任务

## 7.2 通知模板
- 受理：`已受理 TASK-xxx，当前 queued`
- 进度：`TASK-xxx 60%（正在执行单元测试）`
- 完成：`TASK-xxx done，产出路径...`
- 告警：`TASK-xxx failed（原因/建议动作）`

## 9. 实施计划（代码开发导向）

## Phase 1（1周）MVP
1. 实现 Task API + 状态机
2. 接入 2 个 Worker（coding/doc）
3. Telegram 指令与通知打通
4. 上线基本重试与失败告警

## Phase 2（1~2周）稳定化
1. 增加审批流与高风险闸门
2. 增加失联接管与回退路由
3. 增加审计日志与统计面板

## Phase 3（按需）增强
1. 子任务并行 DAG
2. SLA/优先级队列
3. Router 主备容灾

---

## 10. 验收标准

1. 任务成功率 ≥ 90%（首月）
2. 平均任务可见状态更新间隔 ≤ 60s
3. 失败任务告警到达率 100%
4. 高风险动作未经审批执行率 = 0
5. 审计日志覆盖率 100%（关键动作）

---

## 11. 风险与缓解

- 风险：单中枢故障
  - 缓解：每日备份状态库 + 预留热备方案
- 风险：Worker 权限过大
  - 缓解：最小权限、目录隔离、审批闸门
- 风险：任务描述歧义
  - 缓解：入口标准化 + 任务模板化

---

## 12. 结论

在当前约束下，最佳方案为：
**OpenClaw 统一编排（控制面） + OpenCode 多实例执行（数据面） + Telegram 人机入口 + Tailscale 内网通信。**

该方案可快速落地，且对后续扩展到更复杂协同具备清晰演进路径。
